单元测试的艺术

1.基础知识

  • 单元测试的要求
    1. 自动化可重复执行
    2. 容易实现
    3. 有复用价值
    4. 运行方便
    5. 结果稳定
    6. 完全隔离
    7. 完全控制单元
    8. 失败后易于定位问题
  • 集成测试
  • 测试驱动开发（TDD）

2.第一个单元测试

  • 语法

    assertTrue();
assertFalse();
assertEquals();
assertSame();
assertNull();
assertNotNull();
assertNotEquals();
assertNotSame();

  • AssertThat

    public static <T> void assertThat(T actual, Matcher<? super T> matcher) {
    assertThat("", actual, matcher);
}

    Matcher

    https://blog.csdn.net/smxjant/article/details/78206435

    一般匹配符

    is anyOf anything allOf not

    字符串匹配符

    containsString endsWith startWith equalTo equalToIgnoringCase

    数值相关匹配符

    closeTo greaterThan lessThan greaterThanOrEqualTo lessThanOrEqualTo

    集合相关匹配符

    hasEntry hasItem hasKey hasValue

  • @Rule

    https://blog.csdn.net/weixin_34399060/article/details/93886426

    Verifier：所有测试结束后对测试执行结果添加额外的逻辑验证测试最终成功与否。该抽象类为子类提供一个接口方法verify()供扩展

    ErrorCollector：是Verifier类的一个子类实现，用于在测试执行过程中收集错误信息，不会中断测试，最后调用verify()方法处理。

    TemporaryFolder：是抽象类ExternalResource的一个子类实现，用于在JUnit测试执行前后，创建和删除临时目录

    TestName：是抽象类TestWatcher的一个子类实现，用于在测试执行过程中获取测试方法名称。在starting()中记录测试方法名，在getMethodName()中返回

    TestWatcher：监视测试方法生命周期的各个阶段。该抽象类为子类提供了五个接口方法succeeded(), failed(), skipped(), starting()及finished()供扩展

    Timeout：与@Test中的timeout相对应，@Test只能修饰待测试方法，Timeout可以修饰待测试类

    ExpectedException：与@Test中的expected相对应，提供更强大灵活的异常验证功能，@Test只能修饰待测试方法，ExpectedException可以修饰待测试类

3.使用存根破除依赖

概述

存根是对系统中存在的一个依赖项的可控制的替代物。通过使用存根，

如何破除依赖

  1. 找到被测试对象使用的外部接口或API
  2. 将接口的底层实现替换成能控制的内容

重构代码设计以提高可测试性

  • 重构

    不改变代码功能的前提下对代码的修改

  • 接缝

    是代码中可以插入不同功能的地方

  • 重构类型

    可以把具体类抽象成接口或委托；重构使得能够注入委托和接口的伪实现

  • 重构思路

    1. 抽取接口使底层可替换
    2. 依赖注入，在被测试单元注入伪实现
    3. 构造函数注入，在构造函数层注入伪对象
    4. 用伪对象模拟异常
    5. getter,setter注入伪对象
    6. 方法调用前注入伪对象
      • 使用工厂类构造
      • 在发布版本中隐藏接缝
      • 不同的间接层次
      • 伪造方法使用局部的工厂方法

使用模拟对象进行交互测试

  • 基于状态的测试
  • 交互测试

模拟对象与存根的区别

存根永远不会导致测试失败，模拟对象可以

资料查阅理解：伪对象就是后两者的统称，模拟对象是一种功能更强大的存根，模拟对象会记录交互的状态，不记录交互状态的就是存根

个人理解：存根面向特定调用过程的结果，模拟对象注重对象状态

隔离框架

动态生成伪对象

  • NSub

    ILogger logger = Substitute.For();

  • Mockito

    ILogger logger = Mockito.mock(ILogger.class);

模拟值

  • NSub

    IFilenameRules fakeRules = Substitue.For();
fakeRules.IsValidLogFileName("strict.txt").Returns(true);

  • Mockito

    IFilenameRules fakeRules = Mockito.mock(IFilenameRules.class);
Mockito.when(fakeRules.IsValidLogFileName("strict.txt")).thenReturn(true);

隔离框架的优缺点

应避免的陷阱

  • 测试代码不可读
  • 验证错误的内容
  • 一个测试多个模拟对象
  • 过度指定测试

隔离框架设计反模式

  • 避免概念混淆

    模拟对象与存根不可混用

  • 使用Given-when-then模式而非录制重放

  • 避免粘性行为

优秀单元测试的支柱

可靠性、可维护性、可读性

编写可靠的测试

  1. 删除或修改测试

    测试失败的原因：产品缺陷、测试缺陷、语义或API变更、冲突或无效的测试

  2. 避免测试中的逻辑

    避免循环以及控制逻辑

  3. 只测试一个关注点

  4. 分离单元测试与集成测试

  5. 用CR确保单测覆盖率

编写可维护的测试

  • 测试私有或受保护的方法

    • 测试的方法设为public
    • 独立逻辑的方法抽取到新类
    • 使方法称为静态方法
    • 使方法称为内部方法

  • 去除重复代码

  • 以可维护的方法使用setup方法

    1. 避免只在某些测试中使用的初始化对象
    2. 避免冗长难懂
    3. 避免准备伪对象
    4. 停用setup方法

  • 实现测试隔离

    坏味道

    1. 强制的测试顺序
    2. 隐藏的测试调用
    3. 共享状态损坏
    4. 外部共享状态损坏

  • 避免对不同关注点多次断言

  • 对象比较应提高可维护性，如重写toString

  • 避免过度指定

    1. 避免纯内部行为
    2. 在需要存根时使用模拟对象
    3. 不必要的顺序指定或精确匹配

编写可读的测试

  1. 注意单测命名

  2. 变量命名

  3. 有意义的断言

    不要重复测试控制台输出信息

    不要重复测试名中包含的信息

    若无有用信息则无需输出

    提供应发生与未发生的信息

  4. 断言和操作分离

  5. 避免滥用setup与teardown
